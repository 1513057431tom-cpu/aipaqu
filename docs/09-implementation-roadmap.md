# 实施路线

每个阶段完成后必须通过自动测试、构建检查和人工验收，才能进入下一阶段。

## 阶段 0：文档基线

- [x] 产品需求和范围。
- [x] 系统架构与扩展边界。
- [x] 数据模型与状态机。
- [x] API 契约。
- [x] LangGraph 工作流。
- [x] 前端信息架构。
- [x] 安全、部署和运维方案。
- [ ] 用户评审并确认文档基线。

## 阶段 1：可运行基础

- 建立 Nuxt、FastAPI、测试和 Docker Compose。
- 实现配置加载、结构化错误、健康检查和数据库迁移。
- 实现登录、会话、`ADMIN`/`EDITOR` 权限。
- 建立 MySQL、Redis、Elasticsearch 和文件存储适配器。

验收：用户可以登录后台；服务健康检查可定位依赖故障。

## 阶段 2：MVP 采集需求工作台

- 实现 ResearchBrief、来源、材料、数据集和约束实体。
- 实现 DAY/WEEK/MONTH/CUSTOM 时间解析。
- 实现文本、URL 清单、文档和结构化数据上传。
- 完成工作台分步界面、字段校验、上传状态和运行前校验。
- URL 清单在 MVP 中先作为用户资料入库，不要求联网抓取。

验收：用户能够完整描述研究任务，上传资料，并启动一次报告任务。

## 阶段 3：MVP 报告工作流

- 实现 Brief、Preprocess、Research、Analyze、Summarize、Review、Compose、Human Review、Export 节点。
- 实现节点状态、预算、自动返工上限和失败节点恢复。
- 接入 DeepSeek Provider 和 Embedding Provider 的稳定接口；MVP 可使用简单本地检索或数据库检索替代完整混合检索。
- 实现带 `claimId`、引用和来源摘要的报告草稿。

验收：上传资料能够生成带引用的报告草稿；失败可从失败节点恢复。

## 阶段 4：MVP 报告编辑与导出

- 实现手工创建和编辑报告规则版本。
- 实现三栏报告编辑器、引用查看、人工版本保存和单章节重生成。
- 实现提交审核、批准和发布状态。
- 实现 Markdown 和 DOCX 导出。

验收：人工修改不会被非目标章节重生成覆盖；批准后的报告可导出 Markdown/DOCX。

## 阶段 5：采集与知识库增强

- 实现 Collector 接口。
- 依次交付 RSS、API、静态网页和 Playwright 登录态采集。
- 实现 `BrowserRuntimeProvider`、`SiteAdapter`、`CrawlPolicy` 和浏览器 Profile 加密引用。
- 实现采集调试工作台：DOM 快照、选择器测试、Markdown 预览、网络状态摘要和适配器草稿。
- 实现正文解析、时间过滤、三级去重、分块和 Elasticsearch 索引。
- 实现情报中心检索、聚类和来源查看。

验收：指定期间资料可稳定采集，重复内容不会重复入库，网络资料与上传资料共同参与分析；凭据失效、验证码、403/429 和 robots 禁止时能进入可解释失败状态。

## 阶段 6：图表、PDF 与交付增强

- 实现数据集快照、ECharts 编辑、图表渲染和来源映射。
- 实现 PDF 导出。
- 实现 SMTP 配置、测试邮件、正式发送和幂等重试。
- 实现下载、发送和审核审计。

验收：图表数字可追溯；批准后的报告能够稳定导出并发送，重试不会重复投递。

## 阶段 7：系统验证

- 单元测试：日期、去重、状态机、引用和预算。
- 集成测试：数据库、Redis、Elasticsearch、文件、SMTP 假服务。
- E2E：创建需求至邮件发送的关键路径。
- 安全测试：SSRF、恶意文件、提示注入、权限越界和秘密泄露。
- 恢复测试：Worker 中断、模型失败、凭据失效和 Redis 重启。

最终完成标准：产品需求文档中的所有首版验收项均有可重复验证证据。
