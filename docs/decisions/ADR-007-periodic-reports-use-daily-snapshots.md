# ADR-007：周报和月报默认聚合日报快照

## 状态

Accepted

## 日期

2026-07-02

## 背景

周期报告存在两类不同任务：

- 日报：面向当天资料，可以触发采集、解析、检索和分析。
- 周报/月报：面向一段时间内已经形成的日报结论，核心是汇总、对比、去重、趋势判断和二次审核。

如果周报/月报点击后直接复用完整采集工作流，会出现几个问题：

- 用户只是想生成周期汇总，却被意外打开浏览器。
- 周报/月报会重复采集同一来源，增加成本、限流和合规风险。
- 周期报告无法解释“到底使用了哪些日报版本”。
- 某天日报缺失时，系统会把缺失输入误当成需要自动补采，导致不可控行为。

## 决策

周报和月报默认使用 `AGGREGATE_DAILY_SNAPSHOTS` 输入模式：

- 周报读取对应自然周内已审核或已发布的日报快照。
- 月报读取对应自然月内已审核或已发布的日报快照。
- 快照至少包含 `contentJson`、Markdown/HTML 渲染快照、引用摘要、内容摘要和生成时间。
- 周报/月报任务创建时写入 `ReportInputSnapshot`，固定本次使用的日报版本。
- 缺少日报时返回缺失日期清单，任务不得自动进入 `Collect`，不得自动打开浏览器。
- 用户明确选择“补生成缺失日报”或“重新采集来源”时，系统才能创建独立的日报补生成或采集任务。

## 备选方案

### 周报/月报直接重新采集全部来源

- 优点：不依赖日报是否完整。
- 缺点：重复采集、成本高、容易触发限流，且不符合“周期汇总基于日报沉淀”的产品预期。
- 结论：拒绝作为默认行为。

### 周报/月报允许自动补采缺失日期

- 优点：用户操作少。
- 缺点：点击周评/月评可能隐式打开浏览器或访问外部站点，行为不可预期。
- 结论：只能在用户明确确认后创建补生成任务。

### 周报/月报基于日报快照聚合

- 优点：可解释、可审计、成本可控，能稳定复现周期报告。
- 缺点：依赖日报生成质量，需要处理缺失日期。
- 结论：作为默认方案。

## 后果

- 需要保存日报 `ReportVersion` 的 HTML/Markdown 快照和内容摘要。
- 周报/月报任务需要先做周期输入完整性检查。
- 前端点击周评/月评只进入周期汇总配置，不打开浏览器。
- 缺失日报成为显式状态，由用户选择补生成或取消。
