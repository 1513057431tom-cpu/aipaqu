# ADR-006：工作流内核保持框架中立

## 状态

Accepted

## 日期

2026-06-30

## 背景

平台后续可能面向商业部署、私有化交付、不同模型供应商、不同采集策略和更严格的审计要求。LangGraph、LangChain 等框架可以提高早期研发速度，但如果把它们作为业务内核，会带来以下风险：

- 数据库、API 和前端文案泄漏第三方框架概念，未来迁移成本高。
- 商业客户可能要求私有化、可观测、可恢复、可审计和可控成本，而框架默认行为不一定完全匹配。
- 工作流状态、节点输出、人工等待和失败恢复需要长期稳定，不应随外部框架升级而改变。
- 后续可能需要接入 Temporal、Prefect、Celery Canvas、自研执行器或客户已有调度平台。

## 决策

平台定义自己的工作流内核契约，包括：

- `WorkflowDefinition`
- `WorkflowRun`
- `WorkflowNodeRun`
- `WorkflowEngine`
- `WorkflowNode`
- 节点输入输出 Schema
- 检查点、预算、审计和恢复规则

LangGraph、LangChain、Temporal、Prefect 或自研执行器都只能作为 `WorkflowEngineAdapter` 接入。业务实体、REST API、前端展示、审计日志和插件契约不得依赖某个第三方框架的对象模型或专有状态格式。

## 备选方案

### 直接采用 LangGraph/LangChain 作为内核

- 优点：研发启动快，生态示例多，适合验证 Agent 节点编排。
- 缺点：商业化后迁移和定制成本不可控，状态格式和执行语义容易泄漏到平台公共契约。
- 结论：可作为适配器或实验实现，不作为平台内核。

### 直接采用 Temporal 作为内核

- 优点：强恢复、长事务、可观测和生产经验成熟。
- 缺点：部署和学习成本较高，MVP 阶段会拖慢闭环验证。
- 结论：保留为后续适配器选项。

### 完全自研执行器

- 优点：控制力最强，契合平台审计和预算模型。
- 缺点：需要自己实现恢复、并发、调度和可观测能力。
- 结论：MVP 可实现最小自研执行器，但接口必须允许替换成更成熟的执行器。

## 后果

- 早期实现需要先定义节点契约和运行记录，不能直接把框架对象传到底。
- 文档、数据库字段和 API 命名使用 `WorkflowEngine`、`WorkflowDefinition`、`WorkflowRun` 等平台概念。
- 可以在不破坏业务契约的情况下替换执行器。
- 后续选型可以按客户规模、部署环境和审计要求选择不同适配器。
